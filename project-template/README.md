# AI 개발 운영 스캐폴드 (솔로판)

새 프로젝트를 **AI 에이전트와 규율 있게** 개발하기 위한 문서 뼈대다. "코드를 짜기 전에 무엇을 어디에 적어두면 길을 잃지 않는가"를 구조로 굳혔다. 이 폴더를 새 프로젝트에 복사하고 플레이스홀더만 치환하면 시작할 수 있다.

> 원본: CMS 프로젝트의 `CLAUDE.md` + `docs/` 체계에서 **재사용 뼈대만** 추출·추상화(솔로 전제). 현 스택(Spring+React) 예시는 `<!-- 예시(교체) -->`로 남겼다.

---

## 1. 철학 — 왜 이런 구조인가

이 스캐폴드가 강제하는 것은 단 하나: **"열린 것을 머릿속·대화에만 두지 말고, 종류별로 정확히 한 곳에 즉시 적는다."** AI 세션은 컨텍스트가 길어지면 잊는다. 사람도 잊는다. 그래서:

- **기록 강제("말로 끝내지 않는다")** — "한계가 있다/ 이건 나중에 / 이건 정해야 한다"를 **말하는 순간 해당 보드에 적는다.** 응답 말미의 "~해야 합니다"는 증발한다. 보드에 적힌 것만 남는다.
- **loose-end 라우팅** — 열린 것은 성격이 다르다. 성격별로 **다른 보드**에 넣어야 나중에 찾는다(§4 표).
- **TDD = 인수조건이 곧 테스트** — 기능 문서의 Given-When-Then을 테스트로 먼저 옮기고 최소 구현. 문서·코드·테스트가 한 몸.
- **교훈 누적** — 비자명한 버그·함정은 원인→해결로 남겨 재발 방지(`lessons-learned`).
- **refactor seam** — 미확정 결정은 "한 곳만 바꾸면 되는" 지점에 격리해 잠정값으로 진행하고 나중에 최소 변경으로 뒤집는다.

이 규율이 없으면 AI는 그럴듯하게 진행하지만 열린 것들이 조용히 사라진다. 이 폴더는 그걸 막는 장치다.

## 2. 문서 택소노미 — 무엇이 어디에

네 층으로 나뉜다. 각 층은 역할이 다르다.

| 층 | 파일 | 역할 | 성격 |
|---|---|---|---|
| **진입점** | `CLAUDE.md` | 에이전트가 매 세션 첫 줄에 읽는 규칙·인덱스 | 짧고 안정적 |
| **코어** | `docs/00~05` + `90` + `lessons-learned` | 프로젝트 전역 설계·규약·TDD·보류·교훈 | 프로젝트당 1세트 |
| **모듈** | `docs/<module>/00~07` | 기능 묶음 상세(개요·데이터·API·인수조건·에러) | 모듈마다 1세트 |
| **작업 보드** | `working-area/*` | 진행 중 생기는 열린 것들의 기록장 | 계속 쓰고 지움 |

원칙: **코어/모듈 = 기획·정본**(신중히 바뀜), **작업 보드 = 살아있는 메모**(수시로 갱신). 둘을 섞지 않는다.

## 3. 항목별 스펙 — 무엇을·왜·채우는 법

### 진입점
- **`CLAUDE.md`** — *무엇*: 프로젝트 한 줄 정의·스택·문서 인덱스·작업 원칙(TDD·기록강제·seam)·메모리 규칙·git·빌드·autonomous 모드. *왜*: 에이전트가 매번 여기서 규칙을 로드한다. *채우기*: `CLAUDE.template.md` 복사 → 플레이스홀더 치환. **짧게 유지**(원칙만, 상세는 docs로 링크).

### 코어 docs
- **`00-project-overview`** — 시스템 정의·모듈 지도·의존 방향·표준 요청 흐름·범위 경계. *왜*: "이 프로젝트가 무엇이고 API 한 건이 어떤 파이프라인을 타는가"의 한 장.
- **`01-architecture`** — 런타임·레이어·디렉토리 구조(feature-first)·인가 규칙. *왜*: 새 코드를 어디에 둘지의 기준.
- **`02-conventions`** — API·응답 포맷·HTTP status·네이밍·도메인 타입·날짜·**코딩 컨벤션·용어집**. *왜*: 일관성의 단일 출처. (원본의 팀표준을 여기 병합.)
- **`03-error-handling`** — 에러 모델·**refactor seam**·에러코드 카탈로그·FE 에러 처리. *왜*: 에러는 계약이다. 실제 동작과 문서가 어긋나지 않게.
- **`04-non-functional`** — 보안·성능·가용성·로깅·**테스트가능성**. *왜*: "돌아간다" 너머의 품질 제약.
- **`05-tdd-guide`** — Red-Green-Refactor·테스트 레벨·모듈화·인수조건→테스트·**메모리 규칙**·완료 기준. *왜*: 개발 방식의 정본.
- **`90-open-items`** — **기술 보류 보드.** 잠정값으로 진행 가능한 한계·미결. 각 항목에 **재개 트리거**(무엇이 충족되면 처리). *왜*: 미룬 것을 잊지 않게. *채우기*: 그때그때 append, 해소 시 `decision-log`로 이관.
- **`lessons-learned`** — 재사용 교훈(날짜/맥락/원인→해결/적용). *왜*: 같은 함정 재방문 방지.

### 모듈 docs (`_module-template/` → `docs/<module>/`)
기능 묶음마다 한 세트 복사. 00 개요 / 01 아키텍처·컨벤션 / 02 데이터모델 / 03 API 명세 / 04 **기능 요구(Given-When-Then=TDD 기준)** / 05 비기능 / 06 TDD 가이드 / 07 에러코드. *왜*: 기능 상세를 코어와 분리해 모듈별로 자족.

### 작업 보드 (`working-area/`)
- **`decisions-needed`** — *정해야 할 열린 결정*(사람 몫). 결정 질문·선택지 장단·잠정값·영향. *왜*: AI가 임의로 정하면 안 되는 정책·제품·되돌리기 어려운 선택을 붙잡아둔다. → 정해지면 `decision-log`로.
- **`decision-log`** — *확정된 결정 기록*(무엇을 왜 그렇게). *왜*: "왜 이렇게 했더라"의 답. 미래의 나/리뷰어용.
- **`pre-deploy-checklist`** — *운영 배포 갭*(시크릿·DDL·인프라·도메인 등 사람이 할 것). *왜*: "코드는 됐는데 운영이 안 되는" 갭 방지. **go-live 단일 점검 지점.**
- **`schema-change-queue`** (옵션) — DB/외부의존 변경 요청 큐. 운영 DB를 직접 못 만질 때.
- **`review-template/`** — 코드리뷰 구조(요약/백엔드/프론트/교차) + 마커 범례(✅수정/⏸보류/◐부분/📄문서화). *왜*: 셀프 리뷰도 같은 틀로.

## 4. loose-end 라우팅 (핵심 표)

열린 것이 생기면 **성격을 보고 정확히 한 보드**에 적는다:

| 생긴 것 | 어디에 | 예 |
|---|---|---|
| 사람이 **정해야** 할 것(정책·제품·되돌리기 어려움) | `working-area/decisions-needed` | "탈퇴 데이터 파기 정책?" |
| 잠정값으로 **진행 가능**한 기술 보류 | `docs/90-open-items` | "rate limit은 메모리로, 다중 인스턴스 시 Redis" |
| **운영 배포** 시 사람이 할 것 | `working-area/pre-deploy-checklist` | "JWT 시크릿 운영값 주입" |
| **확정된** 결정 | `working-area/decision-log` | "토큰 저장 = 메모리 전용" |
| 재사용 **교훈** | `docs/lessons-learned` | "지속 test DB는 CREATE IF NOT EXISTS가 새 컬럼 누락" |
| DB/외부 **스키마 변경** 요청 | `working-area/schema-change-queue` | "이메일 UNIQUE 제약 추가" |

> 규칙: **"~해야 한다"고 말하는 즉시 위 표대로 적는다.** 응답에만 남기지 않는다.

## 5. 적용 절차

**정본 배포 레이아웃(반드시 이대로 — CLAUDE.md의 경로 참조가 이 구조를 전제):**
```
<프로젝트 루트>/
  CLAUDE.md          ← CLAUDE.template.md 리네임+치환
  docs/              ← 00~05·90·lessons·<module>/
  working-area/      ← docs와 형제(루트 직속). docs/ 밑에 넣지 말 것.
```

1. `project-template/`의 **내용물을 프로젝트 루트로 추출**(폴더째 복사 아님): `CLAUDE.template.md`→루트 `CLAUDE.md`, `docs/`→루트 `docs/`, `working-area/`→루트 `working-area/`. (`README.md`·`PLACEHOLDERS.md`·`first-session-prompt.md`는 메타 — 아래 6번에서 처리.)
2. `CLAUDE.md`의 `{{...}}`를 `PLACEHOLDERS.md`대로 치환. 안 맞는 `<!-- 예시(교체) -->` 블록과 **상단 템플릿 헤더 주석** 삭제.
3. `docs/_module-template/`를 첫 모듈명으로 복사(`docs/<module>/`) 후 `{{MODULE}}`·`{{MODULE_UPPER}}` 채움. **다 쓴 `docs/_module-template/`는 삭제**(안 지우면 잔재 grep을 계속 오염 — 다음 모듈용으로 남기려면 grep에서 제외).
4. 안 쓰는 코어 절 삭제(예: 관리형 DB 아니면 CLAUDE §8·`working-area/schema-change-queue.md` 제거). 코드리뷰를 할 땐 `working-area/review-template/`를 `working-area/review-YYYYMMDD/`로 **복사해서** 쓴다(원본 template은 보존).
5. 잔재 점검(`PLACEHOLDERS.md`의 범용 grep 2종).
6. **메타 파일 정리**: `README.md`·`PLACEHOLDERS.md`·`first-session-prompt.md`는 참고용이라 그대로 둬도 되고, 거슬리면 `.ai-template/`로 모아둔다(삭제해도 무방).
7. 첫 AI 세션: **`first-session-prompt.md`의 ① 블록을 그대로 붙여넣는다.** 에이전트가 인터뷰로 플레이스홀더를 채우고 첫 모듈 문서를 함께 초안한 뒤, 이후 기능마다 ② 블록으로 TDD 진행한다.

## 6. 솔로 ↔ 팀 전환

이 스캐폴드는 **솔로 전제**다(팀 협업·동기화 의례·공유repo·공유항목/충돌 로그 제거). 팀 프로젝트로 쓰려면 아래를 되살린다:

- **`docs/06-collaboration`** 신설 — 팀 코드 우선 원칙 + **동기화 의례**(기능 단위 시작 전/커밋 후 `git pull`) + 충돌 해소 규칙.
- **`working-area/shared-edits-log`** — 팀 소유/공유 파일을 편집했을 때 머지 인지용 기록(무엇·왜·공유 포인트).
- **`working-area/sync-conflict-log`** — `git pull`/머지 충돌 해소 기록.
- **`docs/02`의 용어집·주석 양식을 `07-team-standards`로 분리**해 팀 단일 출처화.
- **loose-end 표에 2줄 추가**: 공유 파일 편집 알림→shared-edits-log / 동기화 충돌→sync-conflict-log.

핵심 규율(기록강제·라우팅·TDD·교훈·seam)은 솔로/팀 공통이다. 팀은 보드 두 개가 더 붙을 뿐이다.
