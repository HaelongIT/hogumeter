# used · 05. 비기능 요구

> 전역 `docs/20-non-functional.md` 위에 이 모듈 특유분만. 방향이며 core V3가 확정한다.

## 보안 / 플랫폼 잣대 (절대 원칙 5)
- **번개**(잣대 ③ 공개 목록 수용): 검색당 목록 폴링 1회, `SiteKind.MARKETPLACE` **하한 10분**
  (`scheduler/policy._FLOOR`, 이미 정의). 설정으로 완화 불가(SEC-08). 지수 백오프·거절(403·429)은
  중지+알림(재시도 아님).
- **당근**(잣대 ② IDC 차단·약관): **자동 fetch 시도 금지**(반자동). 감지는 네이티브 키워드 알림에 위임,
  판단만 URL 평가기로. 평가기의 당근 URL도 자동 fetch하지 않는다.
- **SEC-07 개인정보 배제**: `used_listing_observation`·`raw`에 판매자 uid·동 단위 location·광고 추적 imp_id를
  **담지 않는다**(`parse_bunjang`이 이미 걸러내고 `tests/test_privacy.py`가 잠근다). 담는 것은
  `ad`·`bizseller`·`free_shipping`만.
- **차단 우회 금지**: 프록시·UA 위장 금지. 차단 징후 발견 시 즉시 수집 중단 + `docs/98-field-notes.md`
  기록 + 관리자 알림(BM과 동일).

## 성능
- **폴링 부하 = O(검색 수)**: 매물을 개별 재방문하지 않는다. 검색 목록 페이지 한 번으로 신규·가격변동·
  판매완료·끌올을 **전부** 감지한다(`docs/used/00` 생애주기). 상세 fetch는 **알림 승격 시 1회**만
  (`detail_fetched`=false인 매물), 목록 폴링에서는 하지 않는다.
- 3계층 필터·목록 diff는 **순수 온디맨드 계산** — 1인용·검색당 매물 수십~수백 건 규모라 캐시·사전계산 불요.
- 평가기의 신품 기준가 대비 %는 기존 BM 조회 재사용(1h 캐시, 쿼터 보호). 중고 스냅샷은 마지막 폴링 산출물 재사용.

## 배치 / 정리
- **`used_listing_observation`는 insert-only**(`docs/used/02`) — 목록 diff의 입력 자산이라 덮어쓰지 않고
  쌓는다. 따라서 **무한 성장**한다: 보관 기간·정리(파티셔닝/아카이브) 정책은 **미확정**. 되돌리기 어려운
  데이터 정책이므로 필요해지면 `working-area/decisions-needed.md`로 승격(임의 확정 금지). 그전까지는
  1인용 규모라 방치 수용.
- `Listing.status` 전이(ACTIVE→SOLD/REMOVED)는 목록 diff가 갱신 — 삭제하지 않는다(생애주기 자산).

## 신뢰성
- **목록 diff 결정성**: `listing_id` 자연키로 신규·끌올·소실을 가른다. **비교 키가 환경(로케일·정렬)에
  의존하면 멱등성이 조용히 깨진다** — 스냅샷 정렬·dedupe는 바이트 순서로 못박는다(SQL이면 `collate "C"`,
  CLAUDE.md 축적 규칙). 같은 폴링 사이클의 같은 `listing_id`는 한 번만(스냅샷 단위 dedupe).
- **status 코드표 미실측(Q-44)**: 번개 status 매핑이 잠정(비-`"0"`=SOLD_OUT)이라 **예약중을 판매완료로
  오독하면 조기 알림**. 생애주기 판정이 이 코드표에 종속됨을 응답·로그에 표시(과대약속 금지, 원칙 6).
- **적재 계약 드리프트**: collector가 쓰는 `used_listing_observation` 컬럼 집합과 core가 읽는 계약이
  어긋나면 값이 조용히 사라진다 — **스모크 종단 검증으로 잡는다**(신품 4뷰 필드 검증과 동형, `scripts/smoke.sh`).

## 로깅 / 관측
- **파이프라인 카운터**(OBS-02와 동형): 매 폴링/틱마다 신규·가격변동·판매완료·끌올 **건수**를 낸다.
  **0을 생략하지 않는다** — "폴링했는데 신규 0건"이 사라지면 침묵한 폴링과 구분되지 않는다(CLAUDE.md 규칙).
- 관측 출력은 **산문이 아니라 이벤트(JSON, ASCII 이스케이프)** — 문구를 grep하는 테스트는 문구를 굳힌다.
- 번개 파싱 실패·가격 미검출 스킵 로그(파서 품질 지표). 3계층 필터의 REJECTED 사유(어느 계층에서 탈락).
- 수집 실패·차단 징후는 관리자용 텔레그램 채널(OBS-03).
- **cron 침묵 감지**: 번개 폴링을 주기 작업으로 돌리면 "최근에 성공했는가"를 산출물 나이로 별도 게이트가
  본다(신품 `check-backup-freshness` 계열과 동형). "성공했는가"와 "최근에 성공했는가"는 다른 계약이다.
