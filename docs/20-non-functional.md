# 20. 비기능 요구사항 (NFR)

## 보안 (SEC)
- SEC-01 비밀 관리: API 키·봇 토큰·DB 암호는 환경변수/.env(gitignore). **레포에 비밀 커밋 절대 금지.** 커밋 전 훅으로 스캔(gitleaks).
- SEC-02 웹 접근 통제: 1인용이지만 공개망 노출 전제. 최소 토큰/Basic Auth + HTTPS(Caddy 또는 nginx+certbot). 관리 엔드포인트는 인증 필수.
- SEC-03 텔레그램: 봇은 허용된 chat_id 화이트리스트만 응답. 인라인 콜백에 서명/식별 검증.
- SEC-04 extension ingest: 고정 토큰 인증 + 스키마 검증 + 레이트 리밋. 유입 데이터는 신뢰하지 않고 검증 후 저장.
- SEC-05 입력 위생: 크롤링 텍스트는 전부 비신뢰 입력 — 파라미터 바인딩(JPA), 웹 출력 시 이스케이프(XSS), 크기 상한.
- SEC-06 의존성: Dependabot/주기적 취약점 스캔. 베이스 이미지 고정 태그.
- SEC-07 개인정보 최소화: 중고 판매자 닉네임 등 개인 식별 정보는 표시용 최소한만 저장, 별도 수집·프로파일링 금지.
- SEC-08 크롤링 윤리(잣대의 기술 구현): UA 위장 금지, robots 존중, 사이트별 고정 레이트 리밋을 코드 레벨 강제(설정으로 완화 불가한 하한), 차단 신호(403/429) 감지 시 자동 중지 + 관리 알림 — 재시도 강행 금지.

## 성능 (PERF)
- PERF-01 알림 지연: 핫딜 글 게시 → 텔레그램 발송 p95 ≤ 폴링주기 + 30초.
- PERF-02 네이버 쿼터 예산: 일 호출을 쿼터의 50% 이하로 설계(캐시 1h 필수 경유). 쿼터 소진 접근 시 조회 기능 우아한 축소(캐시 서빙 + 안내).
- PERF-03 웹 API p95 ≤ 500ms(비교·기준가 조회는 사전 계산/캐시 활용).
- PERF-04 DB: (variant, firstSeen), (site, postId), (platform, listingId) 인덱스. 기준가는 딜 유입 시 증분 재계산 또는 요청 시 계산+캐시 — 1인용 규모에서 과최적화 금지.
- PERF-05 수집 부하: 게시판당 1req/min, 중고 검색당 10분 하한. 동시성은 사이트당 1커넥션.

## 신뢰성 (REL)
- REL-01 멱등 ingest: 자연키 UNIQUE + upsert. 재실행·중복 폴링에 결과 불변.
- REL-02 장애 격리: 한 사이트 파서 실패가 다른 사이트 수집을 막지 않는다(사이트별 try/backoff).
- REL-03 재시작 내성: 컨테이너 restart=always, 폴링 커서 영속화. 프로세스 재시작 후 중복 알림 금지(발송 기록 영속).
- REL-04 백업: pg_dump 일 1회 + 보존 7일, 주 1회 오프사이트(S3 등 — 백업만 예외적으로 관리형 허용).
- REL-05 마이그레이션: Flyway 단독, 롤백 스크립트 동반.
- REL-06 파싱 드리프트 감지: 사이트별 파싱 성공률/글 유입량이 임계 이탈 시 관리 알림(사이트 구조 변경 조기 발견).

## 관측성 (OBS)
- OBS-01 구조화 로그(JSON), 상관 ID(딜 단위 추적).
- OBS-02 핵심 카운터: 수집 글 수, 매칭 CONFIRMED/CANDIDATE/REJECTED 비율, 병합률, 알림 발송 수, 큐 적체, API 쿼터 사용량.
- OBS-03 에러·드리프트·쿼터 경고는 텔레그램 관리 채널로(별도 chat).
- OBS-04 헬스체크 엔드포인트(컴포넌트별) + Compose healthcheck.

## 운영 (OPS)
- OPS-01 배포: docker compose up 단일 명령. 환경별 .env 분리(local/prod).
- OPS-02 이식성: AWS 종속 서비스 금지(백업 스토리지 제외). compose 파일만으로 타 호스트 이전 가능해야 함.
- OPS-03 시간대: 저장 UTC, 표시 KST. quiet hours는 KST 기준.
