# 구매 의사결정 보조 시스템 — 기획 확정본 v1.2 (최종 봉인)

> 확정일: 2026-07-04 / 상태: **기획 봉인 — 감사 4회 완료, 열린 결정 0건**
> v1.1: 3차 감사 — 단일 사이트 딜 가중분리, 백필 origin, v1 매칭 탈모델, K_fill 연동식, 방해금지 시간대, 판매완료 알림 범위, 스냅샷 출처 표기, 승격 큐 개발 순서.
> v1.2: 4차(최종) 감사 — 등록 주경로 역전(텍스트 검색+후보 선택), 표본 판정 단위 n 통일, 웹 최소 슬라이스 선개발, 방해금지 보류 알림 상태 병기, 최고 강도 1발 원칙.
> 본 문서는 기획회의에서 잠근 결정만 기록한다. 숫자·규칙만 박으면 되는 항목은 "상세설계 이월"로, 의도적으로 미룬 기능은 "Phase 2 이월"로 분리 기재한다.

## 0. 정체성 (한 줄 정의)

**내가 사려는 물건의 "핫딜 기준 적정가"를 커뮤니티/핫딜 데이터로 산출하고, 현재 판매가가 그보다 얼마나 비싼지(갭)를 제시해 "지금 사도 호구가 아닌지"를 판단하게 돕는 1인용 구매 의사결정 보조 시스템.**

- 핵심 지표는 "정가"가 아니라 **핫딜 기준가**와 **갭**이다. (정가는 상시할인 때문에 무의미한 지표로 공식 폐기)
- 당장은 1인용. 단, 서비스화 가능성을 고려해 코어(수집+기준가 엔진)와 전달(알림/웹)을 분리하고, 확장 시 살아남는 부품(공식 API 기반)을 우선한다.

## 0-1. 핵심 원칙 (모든 설계 결정의 잣대)

1. **정직성**: 표본이 빈약하면 통계 단어("기준가", "median")를 쓰지 않는다. 모든 기준가는 표본 수(n건/m개 사이트/기간)를 상시 동반한다.
2. **판단은 사람, 시스템은 근거**: 사기 여부·최종 구매 판단은 사람 몫. 시스템은 원문 링크, 반응 신호, 갭, 위험 신호를 한자리에 모아줄 뿐이다.
3. **속도·재현율 우선**: 놓치는 것이 오알림보다 나쁘다. 첫 발견 즉시 알림, 틀린 것은 사람이 1초 확인으로 쳐낸다(확인 클릭은 학습 데이터가 된다).
4. **손잡이 배분**: 표시 방식을 바꾸는 손잡이는 사용자에게(설정값), 데이터의 진실을 바꾸는 손잡이는 시스템에(고정값).
5. **플랫폼 접근 잣대**: ① 공식 API 우선 ② 기술적 차단이 있으면 우회하지 않음(프록시·위장 금지) ③ 차단 없는 공개 페이지는 저빈도·개인용으로 수용(약관 회색지대임을 인지). 플랫폼 추가/차단 변화 시 이 잣대로 자동 판정한다.
6. **과대약속 금지**: 자동으로 못 잡는 것(장바구니 쿠폰, 댓글 정정 등)은 솔직히 경계를 긋고 "마지막 한 클릭은 사람 + 원문 링크"로 넘긴다.

## 1. 기능1 — 제품 등록

### 등록 방식
- **주 경로: 제품명 텍스트 입력 → 네이버 쇼핑 검색 API → 후보 목록 → 사용자가 선택(클릭 1회 disambiguation).** LLM 불필요 — 후보 선택이 그 역할을 대체.
- **URL 붙여넣기는 보조 입력**: 다나와/네이버 URL은 숫자 ID(pcode 등)만 담고 제품명이 없어 단독으로 식별 불가(4차 감사에서 실측 확인). URL이 입력되면 추출 가능한 만큼만 검색어 프리필. 해당 페이지 fetch는 하지 않는다(잣대 ②③).
- 옵션 트리 자동 획득 없음. **variant 축(옵션)은 등록 화면에서 사용자가 직접 입력/선택.**
- 자연어 "대충 던지기" 등록(LLM disambiguation)은 Phase 2 편의기능.

### 옵션 모델 (2계층)
- **가격결정 축(price-significant)**: 제조사가 가격을 다르게 매기는 축(용량·모델·연식 등). → **variant 단위로 분포·기준가 반드시 분리.**
- **수요 축(demand-sensitive)**: 정가는 같으나 수요로 시세가 갈리는 축(색상·멤버·디자인 등). → 기본은 **묶어서 한 분포**.
  - **묶음/분리는 사용자가 등록/설정 화면에서 토글로 지정**(기본 묶음). 자동 편차 감지·자동 분리 로직은 만들지 않는다(불필요 판정, 장부에서 삭제).
  - 분리 시 글에서 축 값 판별 불가한 딜은 **"값 미상" 버킷**(라인업 미상 처리와 동일 메커니즘)으로. 미상 버킷은 기준가 계산 제외, 승격 큐에서 사람이 분류.
- **"미지정" 상태는 없음.** 다중 선택으로 통일(전부 선택 = 전체).

## 2. 기능2 — 핫딜 기준가 산출 (시스템의 심장)

### 데이터 소스
- **핫딜 커뮤니티 크롤링: 뽐뿌 / 루리웹 핫딜 / 펨코(에펨코리아) 핫딜** 3개로 시작. (메이저 + 글 정형화 + 교차검증 베이스)
- 수집기는 오픈소스 krepe90/user-hotdeal-bot 골격 최대 재활용(게시판당 분당 1회, 상태 변화 추적 기구현). 우리가 얹는 것: 제품 매칭 + 기준가 합성 + 교차검증.
- 일반 커뮤니티 댓글의 "얼마에 샀어요" 수집은 Phase 2 보조 신호.

### 딜 병합·교차검증
- **DealEvent 병합 조건: 정규화된 제품(variant) + 가격 ±α + 24~48h 윈도우.** 여러 사이트 글을 1건으로 병합.
- 2개 이상 사이트 확인 딜 = full 신뢰, 단일 사이트 = 저신뢰 플래그.
- **단일 사이트 딜의 취급 (포함+가중분리)**: 기준가 median 계산에는 **포함**(재현율·표본 확보). 단 **"역대/기간 최저가" 타이틀과 P25 굿딜라인은 교차검증 딜로만** 산출. 표본 수는 **"n건(교차검증 m건)" 이중 표기.**
- **미상 버킷 글끼리는 "제품 후보군 + 가격±α + 48h"로 잠정 병합·잠정 검증 카운트** 허용. 사람이 분류하는 순간 확정 반영. 소급 알림은 하지 않는다.
- 원문 링크(source_url)는 1급 데이터로 항상 보존·노출. 반응 신호(reaction_score: 댓글수·추천수 등)를 가능한 만큼 수집.

### 가격 정규화
- 단위: **실결제가 + 배송비** (무료배송은 배송비 0으로 표기, 무조건 합산 기준으로 통일).
- 과거 핫딜가는 **as-posted(글에 찍힌 값 그대로)** 분포에 넣는다. 카드/쿠폰의 역산·복원은 시도하지 않는다(사람도 못 하는 일).
- 저장 구조: `headline_price`(필수) / `base_price`(분해 가능 시) / `applied_conditions[]`(태그: 카드명·쿠폰명) / `confidence`.
- **딜 대표가격 = 첫 관측가(price_first).** price_min, price_last를 함께 저장(참고용). 분포에는 일관되게 price_first만 사용.

### 이상치 처리 (양방향 분리)
- **위로 튄 이상치**(약정 실질가·끼워팔기 등): 기준가 계산 제외, 알림 없음.
- **아래로 튄 이상치**(가격오류 대박딜 또는 사기): **알림 최우선(🔥 별도 강조) + 기준가 계산 제외 + 갭 극대 표시.** 대박딜은 "알릴 이벤트"지 "평소 기준"이 아니다.
- 이상치 판정은 median/IQR 기반 통계 컷(제품군 특화 로직 없음). 기본 숨김 + **사용자 토글(이상치 포함/숨김)**.
- 사람이 아래 이상치 딜을 보고 **"진짜였다 → 표본 포함" / "사기·낚시 → 영구 제외"** 승격/기각 (승격 큐).
- **약정/성지 등 오알림 방지: 제외 키워드**(전역 기본값 + 제품별 추가, 예: "약정, 번이, 성지, 0원, 요금제")로 알림·기준가에서 배제. "제외" 대신 "⚠️의심 라벨만" 토글 옵션 제공.
- **사후학습**: 오알림 "무시" 클릭 시 해당 글 빈출 키워드를 뽑아 "제외 목록에 추가할까?" 제안. 승격 큐와 동일 메커니즘.

### 기준가 산식 (사용자가 기간 P 설정 시 출력)
1. 기간 내 **최저 핫딜가** + 날짜 (교차검증 딜 중 min)
2. **핫딜 기준가** = 교차검증·이상치 제거 후 **median**
3. **굿딜 라인** = 동일 집합의 **P25**
4. **최근 핫딜** (날짜·가격·출처 사이트)
5. **현재가** (네이버 최저가)
6. **갭** = 현재가 − 기준가, 현재가 − 최저 핫딜가 (%, 원 병기)
7. **신뢰도** = n건 / m개 사이트 / 기간
- **시간 가중 없음**(균등 median). 기간 P 자체가 시간 손잡이.

### 표본 정책 (3단 신뢰 표시)
- **판정·채움 단위 = n (전체 유효 딜: 교차검증 + 단일 사이트 라이브 + 백필 포함, 이상치 제외).** m(교차검증 건수)은 판정에 쓰지 않고 **신뢰 표시 전용** — "n건(교차검증 m건)" 이중 표기로 정직성 유지. K_fill 자동확장도 n 기준.
- **충분** (n ≥ K_display, 기본 5): 기준가+P25 정식 표시. (단 P25·최저가 타이틀은 교차검증 딜로만 산출 — 3차 확정 유지)
- **빈약** (1~4건): 통계 단어 금지. **"최근 핫딜 사례" 나열만**(가격·날짜·출처).
  - 알림 잣대: **보유 사례 중 최저가** 기준(그보다 싸면 강, 비슷하면 약) + "※ 표본 N건, 참고용" 딱지 상시.
  - 오염 방어: median 부재 구간이므로 **현재가 기준 폴백 컷**(비상식 구간은 사례 나열·최저가 잣대에서 잠정 제외 + 승격 큐행).
- **없음** (0건): "핫딜 관측 안 됨 + 현재가만 + 관측 시작" 표시.
  - 대박딜 폴백: **현재가 대비 큰 폭(예: 30%↓)일 때만 "기준 미확립·참고용" 딱지로 알림.**
- **K_display**: 사용자 설정값(기본 5, 범위 3~10). / **K_fill = max(7, K_display + 2)**: 기간 P 내 표본 부족 시 기간 자동 확장으로 채우는 목표 건수(시스템 연동식 — 불변식 K_fill > K_display를 설정 전 구간에서 보존). / **자동 확장 과거 상한 = 12개월**(전역, 카테고리별 조정은 Phase 2).
- 콜드 스타트: 신규 등록 제품의 한계 수용. 가능한 사이트만 과거 글 백필.
- **백필 딜 취급**: DealEvent에 `origin(live/backfill)` 플래그. 백필 가격은 **현재 글에 적힌 값(as-shown)으로 수용**하되 백필 표기(첫 관측가 의미론이 성립 안 함을 인지). **백필 딜에는 교차검증 요건 미적용**(단독이어도 기준가 포함 — 시간 경과로 낚시 글은 걸러졌을 확률 높음, median의 강건성으로 방어). 라이브 딜이 쌓이면 자연 희석.

## 3. 기능3 — 감시·알림

### 수집 구조
- **중앙 수집**: distinct 제품 단위로 1벌 수집, 사용자별 재긁기 금지.
- 사용자 "시간대 설정"은 **방해금지 시간대(quiet hours)로 재목적화** — 설정 구간의 알림은 보류 후 종료 시점에 몰아서 발송. 단 **대박딜(🔥 아래 이상치) 알림은 방해금지를 관통**한다. **보류 알림은 발송 직전 딜 상태를 재확인해 종료 시 "(종료됨)" 표기·가격 변동 시 최신가 병기 후 그대로 발송**(정보 손실 방지 — "놓친 것도 알려줘" 철학). (원요구의 "시간대" 니즈를 즉시성 원칙의 사용자 예외로 부활시킨 것. "알림 평가 주기" 해석은 상시하락 알림 폐기로 대상이 소멸하여 삭제.)
- **알림 단일화 — 최고 강도 1발 원칙**: 한 딜이 복수 트리거를 동시 충족하면 우선순위(🔥대박딜 > 특가(P25/역대최저) > 목표가 충족 > 괜찮은 딜)로 **가장 높은 것 하나만 발송**, 본문에 나머지 충족 조건 한 줄 병기.
- 현재가(네이버)는 알림 트리거가 아니라 **갭 표시용 컨텍스트**: 웹에서 제품 상세 열 때 온디맨드 조회 + 1시간 캐시. 상시 가격하락 알림은 폐기(핫딜 목적과 불일치).

### 알림 트리거 (신품)
- **1순위 = 새 핫딜 이벤트.** 발화 조건: **[기준가 이하 → "괜찮은 딜" / P25 이하 또는 역대최저 근접 → "특가"] OR [사용자 목표가 이하]** (목표가는 제품별 선택 입력, 병존 OR).
- **첫 발견 즉시 알림**(미검증이라도). 첫 알림에 **검증 상태(단독 발견/N개 사이트) 상시 표기.**
- **후속 알림**: ① 교차검증 달성 시 "✅ N개 사이트 검증됨" ② 본문 가격 변동(재풀림·인하·인상) ③ 품절/종료. **댓글 정정은 제외**(원문 링크로 대체).
- 알림 채널: **텔레그램 봇.** 승격 큐 액션(맞음/아님/무시) 인라인 버튼 미러링 방향 채택(상세는 상세설계).

### 사람 승격 큐 (웹)
- 미상 라인업/축값 분류 + 아래 이상치 승격/기각 + 사후학습 키워드 제안 수락을 **하나의 큐**로 통합. 안 치워도 시스템은 안전하게 돌고(보수 기본값), 치우면 정확도 상승. 분류 패턴은 규칙으로 저장되어 미상 큐가 시간이 갈수록 마른다.
- **개발 순서**: 1차 검증 단계의 승격 액션은 **텔레그램 인라인 버튼으로 먼저 커버**, 웹 큐 화면은 그 이후(웹 후순위 방침과의 긴장 해소).

### 매칭 (핫딜 글 → 제품/variant)
- **3단: 자동 확정 / 후보(사람 1초 확인) / 자동 기각.** 재현율 우선 — 알림 임계는 후보 단계가 흡수, 정밀도는 확인 클릭이 보장.
- **v1 매칭 수단 = 탈모델**: 정규화 문자열 매칭(토큰화 + 별칭 사전 + 숫자/용량 패턴) + 3단 큐. 임베딩·LLM 등 모델류 미사용 — **임베딩 후보 축소는 Phase 2 승격 후보로 명시.** 확인 클릭은 규칙/별칭 사전으로 축적되어 정확도 상승.

## 4. 기능4 — 구매 시점 판매처 비교

### 정체
- **"지금 사기로 한 순간, 어디서 사는 게 실결제가 최저인가"의 온디맨드 판매처 횡비교.** (핫딜 진위 대조는 부산물로 자동 획득 — 핫딜가를 현재 최저가 옆에 병치)
- 같은 화면에 **기능2의 기준가 갭 병기** → "어디서 + 지금 사도 되나"가 한 화면에.

### 데이터 소스·범위 (v1)
- **네이버 쇼핑 API**(여러 몰 최저가 우산) + **쿠팡**. 개별 몰(11번가·지마켓 등) 직접 크롤링 없음.
- **쿠팡 확보 사다리(확정)**: ② **브라우저 확장(크롬) = 정식 1차 경로** — 사용자가 쿠팡 페이지를 열면 확장이 가격·혜택을 읽어 서버로 전송(variant 매칭 원천 해소, 차단·약관 자유, 가격 이력 부산물). / ① 파트너스 API는 **사다리에서 제거**(회색지대+15만원 문턱, 존재 이유가 ②에 흡수). / ③ **반자동 복붙 = 폴백**(확장 개발 전·고장 기간). / ④ **핫딜 경유 간접가 상시 병기.**
- 확장 개발 시점: 코어 완성 후 **기능4 착수 단계**에서. (첫 삽 아님)
- 기준가 파이프라인용 쿠팡은 여전히 간접(핫딜 글) — 용도별 이중 트랙.

### 비교 규칙
- **실결제가 + 배송비**로 통일(기능2 정규화 규칙 적용 선언).
- **와우가/일반가 둘 다 저장·표시** + "내 기준" 보기 필터.
- 결과 캐시 1시간.
- **약속 수위(명문화)**: 완전 자동 체감가 아님. "1차 자동 필터 + 페이지에 노출되는 공개 혜택 표시 + 판매처 직링크, 장바구니 쿠폰·카드 청구할인 등 결제단계 혜택의 최종 확인은 사람."

## 5. 기능5 — 중고

### 성격
- **기준가 합성을 하지 않는다.** 중고는 상태 편차 때문에 분포 합성이 무의미. → **정밀 조건검색 + 목표가 알림 + 판단 보조.**
- **UsedSearch는 기존 Product에 종속된 서브 개체**(등록 흐름: 제품 먼저 → 그 아래 중고 검색 추가).

### 매칭 3계층
- **필수 (AND)**: 전부 매칭돼야 후보 (제품 정체성: "아이폰17", "256").
- **가산 (OR 그룹)**: 그룹별 모드 토글 — **A(정렬/배지용: 없어도 알림, 있으면 상단+배지)** / **B(트리거: 그룹 중 1개 이상 있어야 알림)**. 동의어는 사용자가 그룹 안에 직접 나열("S급, 에스급, 민트")— 시스템 내장 동의어 사전 없음.
- **제외 (NOT)**: 전역 기본값("부품용, 액정파손, 침수, 삽니다, 매입" 등) + 제품별 + 사후학습 제안(신품과 동일 메커니즘).

### 수집 구조
- **검색 목록 페이지 폴링** — 매물 개별 재방문 아님. 부하 O(등록 제품 수). 한 번의 목록 요청으로: 새 매물 / 가격 변동 / 판매완료·삭제(목록 소실) / 끌올(매물 ID dedupe로 중복 알림 방지) 전부 감지.
- 폴링 주기: **기본 10분, 제품별 조정 가능.**
- **상세 페이지 fetch는 "알림 승격" 시 딱 1회**(판매자 설명 확보). 본문만 수정된 변화(가격 불변)는 추적하지 않음(의도적 포기 — 목표가 판정에 무영향).
- 생애주기 알림: **가격 변동·판매완료 모두 알림 발송**(신품 후속 알림 정책과 일관). 단 **판매완료 알림은 "알림이 나갔던(승격된) 매물" 한정** — 스냅샷 전체 매물에는 미적용(스팸 방지).

### 플랫폼 전략
- **번개장터 = 풀 자동 파이프라인** (전국 검색 가능, 목록 접근 수월 — 잣대 ③ 수용).
- **당근 = 반자동** (IDC 대역 차단 + 약관 명시 금지 + 지역 구조 → 잣대 ② 우회 금지):
  - 감지: **당근 네이티브 키워드 알림에 위임** (실시간 푸시, 합법).
  - 판단: **URL 평가기** — 입력 3단 폴백: ① URL fetch 시도(단건은 열릴 수도) → ② 텔레그램에 페이지 텍스트 복붙 → ③ 가격·지역 수동 입력. 입력 계층은 처음부터 추상화("URL 또는 텍스트").
- **중고나라 = v1 제외** (네이버 카페 구조).
- 참고: AWS든 오라클이든 IDC 대역은 동일하게 당근 차단 대상 — 호스팅 변경으로 해소 불가, 자동 fetch는 시도하지 않는 것이 확정 원칙.

### URL 평가기 (범용 매물 판단 입구 — 플랫폼 무관)
출력 3종 (상태 요약은 제거 — 원문 10초 독해로 충분, 판단 경로에서 하는 일 없음):
1. **가격 맥락**: 신품 핫딜 기준가 대비 % + **"현재 활성 매물 스냅샷"**(폴링 중인 동일 검색의 활성 매물 가격을 **통계 가공 없이 나열**, 평가 대상이 그중 어디쯤인지 표시. "시세 합성" 아님 — 성격 결정과 정합). 스냅샷은 번개장터 폴링 산출물이므로 **화면에 "비교 대상: 번개장터 활성 매물" 출처 상시 표기**(당근 매물 평가 시 직거래/택배 맥락 차이 인지용).
2. **위험 신호**: NOT 키워드 계열(업자 레퍼토리: "이민·결혼 급처" 등, 선입금 유도) + "스냅샷 최저가보다 X% 이상 쌈" 플래그. 판정 없이 신호 나열만.
3. **원문 링크 + 구조화 필드**(가격·지역·등록일).
- LLM 미사용(v1). 단 **추출 부분을 인터페이스로 분리**해 v2에서 규칙→LLM 교체 가능하게(확장 여지 명시).

### 메모·비교 (하이브리드)
- **자유 메모** 기본(마찰 최소, 글에 없는 관찰 포함 가능).
- **비교축은 제품 단위로 사용자가 정의**("배터리%", "구성" 등 — 제품마다 중요한 축이 다름).
- 메모 값 → 축으로 **승격**(선택적, 비교가 필요해진 시점에).
- **병렬 비교표**: 상단 축 정렬(빈칸 노출 = "확인 안 한 항목" 체크리스트 겸용) + 하단 매물별 자유 메모 전문.
- 메모→축 자동 추출은 Phase 2(LLM 자리).

## 6. 도메인 모델 (핵심 개체)

- **Product** (canonical 제품) — 가격결정 축 / 수요 축(묶음·분리 토글) 정의 포함
- **Variant** (가격결정 축 조합 단위 — 분포·기준가의 기본 단위)
- **DealEvent** (병합된 핫딜 1건): normalized_product/variant, headline_price, base_price?, applied_conditions[], confidence, price_first/min/max/last, shipping, source_sites[], source_url, reaction_score, confirmed_count, first_seen/last_seen, status(신규→검증→변동→종료), 이상치 플래그(방향), 미상 여부
- **PriceHistory** (현재가 시계열 — 온디맨드 갱신)
- **Alert** (제품별: 목표가?, 기간 P, K_display, 제외 키워드, 알림 설정)
- **UsedSearch** (Product 종속: 3계층 키워드, 폴링 주기, 목표가)
- **Listing** (중고 매물: 플랫폼, ID, 가격 이력, 상태)
- **listing_note / comparison_axis / listing_axis_value** (메모·축 — EAV 3테이블)
- DB: **PostgreSQL 단일.** 축·메모는 EAV 정식 테이블(JSONB 남용 금지, JSONB는 크롤링 원본 보관용으로 한정).

## 7. 아키텍처·스택

- **코어**: Spring Boot (기준가 엔진, 알림, 매칭, 웹 API) — 상시 가동.
- **수집기**: **Python 별도 컨테이너** (krepe90/user-hotdeal-bot 최대 재활용 + 번개 폴러). 코어와는 PostgreSQL(또는 REST)로만 접합 — **폴리글랏 확정.**
- **웹**: React (등록·조회·비교·승격 큐·설정) — 단계적: **최소 슬라이스(제품 등록 + 후보 선택 + variant/키워드/목표가 설정 화면)는 1차 검증 전에 선개발**, 조회·비교·큐 화면은 후순위(초기 승격 액션은 텔레그램 버튼). 1차 전달은 텔레그램.
- **알림**: 텔레그램 봇 (+ 인라인 버튼).
- **확장**: 크롬 확장 (기능4 단계에서, 쿠팡 리더).
- **호스팅**: **AWS** (학습 목적 우선). 단 **이식 가능하게** — Docker 컨테이너 패키징, AWS 종속 관리형 서비스에 코어 로직을 얹지 않음(EC2+컨테이너+Postgres 자기완결). 프리티어 종료 시점에 잔류/이전 재결정.
- 번개 폴링 쿼리 전략: **넓은 쿼리(제품 대표어) + 3계층 필터는 로컬 적용** (재현율을 플랫폼 검색엔진에 위임하지 않음).

## 9. 1차 검증 계획

- 검증 제품: **아이폰 17 일반 256GB (자급제)** — 약정 오염·라인업 미상·표본 빈약이 전부 터지는 최적 시험대.
- 수집 사이트: 뽐뿌·루리웹·펨코 3개.
- 성공 판정: 산출된 기준가가 사용자 체감 시세와 부합 + 오알림이 승격 큐/키워드 학습으로 수렴하는지.

## 10. 상세설계 이월 (숫자·규칙 박기)

±α(병합 가격 허용폭) / 병합 윈도우 확정(24 vs 48h) / 이상치 컷 수치(IQR 배수 등) / 콜드스타트 대박딜 폭(30% 가안) / reaction_score 사이트 간 정규화 / 백필 가능 깊이(사이트별 실측) / 웹 접근 통제(공개망 노출 시 최소 인증) / 제품 노화·만료 처리 / 텔레그램 인라인 버튼 상세 / DealEvent 상태기계 전이 다이어그램.

## 11. Phase 2 이월 장부

자연어 등록(LLM disambiguation) / **신품 매칭 임베딩 후보 축소(v1 탈모델의 승격 후보)** / 댓글 기반 실구매가 신호 / 중고 상태요약·추출 LLM 교체 / 메모→축 자동 추출 / 카테고리별 과거 상한 / 색상별 가격 편차 자동 감지 없음 — 필요 시 사용자 토글로 충분(재검토 불요 확정) / 쿠팡 가격 이력 축적(확장 부산물) / 서비스화 시: 데이터 계약 재검토(네이버 쿼터·크롤링 컨테이너 대체·법무 검토) 필수 게이트.

## 12. 리스크 인지 장부 (수용한 회색지대)

- 커뮤니티 크롤링: 약관 회색지대, 저빈도·개인용으로 수용. 서비스화 시 재검토 게이트.
- 번개장터: 현재 차단 없음 전제. 차단 도입 시 당근과 동일하게 반자동 격하(잣대 자동 적용).
- 당근 URL 단건 fetch: 1단 폴백으로 시도만, 실패 전제 설계.
- 크롬 확장: 쿠팡 페이지 구조 변경 시 유지보수 발생 상존.
- AWS 프리티어 12개월 종료 후 상시 비용 — 이식 설계로 헤지.
