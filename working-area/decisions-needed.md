## D-3. 차단(403/429)으로 자동 중지된 사이트의 재개 경로

> 등록 2026-07-09 (collector 폴링 스케줄러 착수 중 발견, `docs/91` Q-39에서 승격).

**무엇을 정해야 하나**: SEC-08에 따라 차단 신호를 받으면 스케줄러가 해당 사이트를 자동 중지한다(`stopped=true`). **사람이 이걸 어떻게 다시 켜는가**가 정해져 있지 않다.

**왜 사람이 정해야 하나 (AI 임의 확정 금지)**
- 구현 선택이 아니라 **확정 원칙의 해석**이다. SEC-08(`docs/20` line 11)의 "차단 신호 감지 시 자동 중지 + 관리 알림 — **재시도 강행 금지**"가 *쿨다운 후 자동 1회 재시도*까지 금지하는지가 갈린다.
- 절대 원칙 5(기술적 차단은 우회 금지)와 직결된다. 잘못 정하면 플랫폼 잣대를 어긴다.
- 잠정값으로 밀고 갈 수 있는 성격이 아니라 `docs/91`이 아니라 이 보드다.

**왜 지금인가 (시한이 있다)**
- 현재 커서는 **메모리**에만 있어서 프로세스 재시작이 우연히 재개 역할을 한다. 이건 설계가 아니라 사고다.
- **`docs/91` Q-59(커서 영속화, REL-03)를 착수하는 순간 `stopped=true`가 디스크에 남아 차단당한 사이트가 영구히 죽는다.** → **Q-59 착수 전에 반드시 결정.**

**선택지**

| | 안 | 성격 |
|---|---|---|
| A | 텔레그램 봇 명령 `/resume <site>` | 사람이 상황 확인 후 명시적 재개. SEC-08과 가장 정합. 단 봇 토큰 미발급(Q-20)에 종속 |
| B | 설정 플래그 / DB 행 수동 수정 | **AI 권고** — 최소 구현, 1인용에 충분, 외부 의존 0, 되돌리기 쉬움. 나중에 A를 얹을 수 있다 |
| C | 쿨다운 후 자동 1회 재시도(예: 24h) | 429는 `Retry-After` 존중이 웹 표준이라 논쟁 여지가 있으나 SEC-08 문구와 충돌 소지. **AI가 임의 채택하지 않음** |
| D | 영구 중지, 재개 없음 | 현 코드의 동작. 영속화 후엔 "한 번 차단되면 끝" |

**연결**: `docs/91` Q-59(커서 영속화, REL-03)·Q-20(봇 토큰). 코드 seam = `collector/src/collector/scheduler/policy.py`의 `advance()` stopped 분기 1곳.

## D-4. HTTPS 종단 방식 (SEC-02 나머지) — 보안 정책이라 임의 확정하지 않음

**무엇이 걸렸나**
- `pre-deploy §C`가 **[필수]**로 남긴 마지막 보안 항목. Basic Auth는 이미 붙었지만 **평문 HTTP에선 자격증명이 그대로 노출**된다. 지금 상태로 공개망에 열면 인증이 있으나 마나다.
- 정지조건 "보안 정책"에 해당한다. 방식 선택이 인프라 형태·시크릿·갱신 책임을 바꾼다.

**선택지**

| | 안 | 성격 |
|---|---|---|
| A | **Caddy를 앞단에** (`tls` 자동 발급·갱신) | **AI 권고** — 설정 3줄, 인증서 갱신 자동. 컨테이너 1개 추가. 로컬은 `tls internal`로 리허설 가능 |
| B | nginx + certbot | 지금 web이 nginx라 서비스 추가 없음. 갱신 cron·볼륨 관리는 사람 몫 |
| C | Cloudflare Tunnel / 리버스 프록시 SaaS | 도메인·계정 종속. 이식성(OPS-02)과 상충 |
| D | HTTPS 없이 VPN·SSH 터널로만 접근 | 1인용이면 성립. 공개망에 안 여는 대신 접근 편의를 버린다 |

**필요한 것**: 도메인 보유 여부, 공개망 노출 의사. 둘 다 사람만 안다.
**연결**: `pre-deploy §C`. 코드 seam = `docker-compose.yml` + `web/nginx.conf` 1곳.

## D-5. 0원(무료) 딜을 기준가 표본에 넣을 것인가 — 데이터 진실이라 시스템 고정

**무엇이 걸렸나**
- `price.py`는 `(무료/무료)`·`GOG 무료` 같은 딜을 **`None`(스킵)** 처리한다. fixture 69건 중 3건.
- 0원을 median 표본에 넣으면 기준가가 무너지고, Tukey 하한을 뚫어 **🔥 최우선 오알림**이 될 수 있다. 반대로 계속 버리면 "무료 배포"라는 **가장 좋은 딜을 영원히 못 본다**(절대 원칙 3: 놓침 > 오알림과 정면 충돌).
- 원칙 4상 **데이터 진실을 바꾸는 규칙**이라 사용자 손잡이가 아니라 시스템 고정이고, 사람이 정해야 한다.

**선택지**

| | 안 | 성격 |
|---|---|---|
| A | 스킵 유지 (현재) | 기준가는 안전. 무료 딜은 영원히 안 보인다 |
| B | 가격 표본에서는 제외하되 **알림·표시에는 태운다**(`price=0` 별도 종류) | **AI 권고** — pricingSet과 signalSet 분리(docs/03)에 정합. 산식 오염 없이 놓침도 없다 |
| C | 표본에 0으로 포함 | median·P25가 무너진다. 권고하지 않음 |

**연결**: `docs/91` Q-18(2). 코드 seam = `collector/src/collector/pipeline/price.py` 반환값 1곳 + core의 pricingSet 필터.

## D-6. 루리웹 잘린 제목의 가격을 복구할까 — 상세 페이지 fetch 여부 (2026-07-22 실측 후 제기)

- **문제(실측)**: 루리웹은 목록에서 제목을 자른다(`...`). 첫 실 사이클 29건 중 **9건(31%)이 잘렸고, 그중 5건(17%)은 가격이 숫자 중간에서 끊겨 유실**됐다(`/ 115...`, `3,14...`). 가격이 없으면 `skippedNoPrice`로 빠져 **딜이 안 만들어지고 알림도 안 간다** — 등록 제품 특가가 긴 제목이면 통째로 놓친다. 뽐뿌는 잘림 0%라 이 문제는 **루리웹 한정**이다. (근거: `docs/98` 2026-07-22)
- **왜 사람이 정하나**: 복구하려면 **상세 페이지를 추가로 fetch**해야 하고, 그건 외부 요청 수를 늘린다 — 절대 원칙 5의 "저빈도·개인용 수용"과 맞바꿈이라 정책 선택이다.
- **선택지**:
  - **① 잘린 것 전부 상세 fetch** — 사이클당 1 → ~10 요청(루리웹). 커버리지 최대, 부담도 최대.
  - **② 등록 제품 별칭이 걸리는 잘린 제목만 상세 fetch** — 사이클당 대개 **0~1건**. 우리가 알림 낼 딜의 가격만 복구하므로 **목적에 정확히 부합하고 가장 싸다.** ⭐추천
  - **③ 안 한다** — 루리웹 긴 제목 딜은 포기(현 상태). 놓침을 받아들인다.
- **추천 근거**: ②는 "우리가 관심 있는 딜"에만 비용을 쓴다. 관심 없는 딜의 가격은 애초에 필요 없다(기준가 표본은 등록 제품 것만 쓴다). 부담 증가가 사실상 없어 원칙 5와 충돌하지 않는다.
- **정하면**: `raw_deal_post.body_text`(Q-18, 현재 죽은 컬럼)가 그 저장 자리다. 죽은 컬럼 allowlist에서 그 줄이 만료된다.
